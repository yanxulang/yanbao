# 言包架构

## 职责分层

`yanbao`是工作流前端，负责参数、清单编辑、事务回滚和人类/JSON 输出。`yanxu::package`是格式与解析核心，负责发现并校验`言序.toml`、选择路径/Git/索引依赖、维护缓存以及写入`言序.lock`。`yanxu`二进制仍负责解释、检查和按权限运行入口。

这种分层保证以下不变量：

1. 包清单与锁文件只有一套解析语义；
2. 言包升级不能偷偷改变运行时依赖选择；
3. 旧脚本可继续使用`yanxu 包`兼容入口；
4. 编辑器和 CI 可依赖版本化 JSON 输出，而不解析人类文本。

## 清单事务

`add`和`remove`开始前保存清单与锁文件字节。言包用同目录临时文件替换清单，再调用核心重新解析。任一步失败都会恢复两份原文件；成功后才报告变更。`--no-lock`是显式例外，只写清单，适合注册表暂不可用时准备元数据。

## 兼容性

言序 1.0 接受惯用中文裸键。言包内部将其规范化为可编辑的 TOML 文档，写回时保持 1.0 运行时可读；核心 1.1 同时修复了标准 TOML 引号中文键的读取。言包 0.1 仍以`yanxu v1.0.0`为最低兼容基线。

JSON 输出以整数`schema_version`独立演进。0.1 的 schema 版本为 1；新增可选字段不提升主版本，删除或改变既有字段语义必须提升。

## 信任边界

- 路径依赖必须由核心规范化并校验包名、入口和版本；
- Git 与注册表制品必须由核心固定修订或校验和；
- `--offline`不得发起获取请求；
- 言包不保存注册表令牌，0.1 也不实现发布；
- `run`仅调用用户安装的`yanxu 包 运行`，权限来自包清单。

注册表发布、凭据隔离、签名、撤回和供应链审计必须先完成威胁模型与协议评审，不能只增加一个上传命令。
