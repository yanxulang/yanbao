# 言包架构

## 纯言序命令层

`src/命令行.yx`统一命令别名、位置参数、旗标、单值选项、互斥关系、透传边界和稳定参数诊断；`src/主.yx`只在解析成功后进入工作流、回滚和用户输出，`src/核心.yx`封装进程调用及工程协议。三个跨平台启动器只定位 `yanxu` 与入口文卷，不解析参数、不修改清单，也不包含依赖业务。

源码开发与正式发行包都要求用户安装言序 1.1.17 或更高版本。这是支持分操作长超时、终止后子进程回收和事务恢复的最低运行时基线。发行包只包含由纯言序源码编译的独立言包应用，不复制工具链运行时。发布流程把`src/主.yx`和同级模块复制到清单目录之外，再按直接 CLI 入口编译；这会显式采用言序的宿主权限兼容策略，使言包能够管理用户通过参数指定的任意工程路径。启动器优先使用`YANXU_BIN`，否则使用`PATH`中的`yanxu`；工具链缺失时立即给出安装提示并终止。

## 唯一依赖语义

言包通过 `yanxu package protocol <JSON>`调用协议版本 1。协议握手公开言序版本、目标平台以及支持的清单、锁、字节码、YXB 和原生 ABI 版本。以下能力只在言序核心实现：

- 格式 1/2 清单读取与格式 2 写入；
- 依赖别名、开发依赖、公开导出和包子模块；
- 直接及传递依赖解析、身份、锁定和离线校验；
- YXB 与独立应用构建；
- 原生制品平台、ABI、校验和和权限门禁。

言包不得自行解析 TOML、选择版本、读取注册表索引或拼装锁文件。

## 工具链兼容通道

CI 从`言序.toml`读取最低支持版本，并固定构建该稳定标签；同时查询言序最新正式 Release，使用公开安装器和平台制品复现普通用户环境；开发联调通道则构建言序`main`。三个通道都读取`yanxu version --json`和工程协议握手，按清单、锁、YXB、原生 ABI 与操作能力判定兼容，不把开发分支绑定到固定版本字符串。

当前稳定 Release 还要完成 Linux、macOS、Windows 的 x86-64 与 ARM64 六目标独立应用、启动器、言窗工程和 Bundle 验收。最低兼容与开发联调失败时同样阻止合并和发布。

## 发布供应链

`scripts/generate-sbom.sh`从唯一清单与锁文件生成确定性的 CycloneDX 1.5 SBOM，公开言包 MIT 许可、最低兼容言序运行时、源码提交、清单格式及摘要、锁格式及摘要。脚本不读取临时目录或构建机绝对路径；同一提交重复运行必须逐字节一致。

`scripts/generate-build-metadata.sh`在六个目标架构的本机运行器上验证正式言序编译器的版本、提交、构建模式和目标，再运行独立应用的结构化版本接口。每份元数据绑定源码引用、提交与时间、目标平台、归档摘要、独立应用摘要、清单和锁文件摘要；编译器目标不一致时拒绝生成结果。

Release 汇总十八份平台资产后生成 SBOM，并为归档、校验文件、构建元数据和 SBOM 创建来源证明，同时为六份归档创建 SBOM 证明。二十一份正式资产只有在摘要、文件清单、元数据和证明全部通过后才进入公开 Release；三系统安装冒烟结束后再从公开 Release 下载复验，最后才设置为稳定最新版。构建或验收失败会保留草稿或撤销稳定资格。

## 结构化输出

`src/输出.yx`是人类文本、JSON 和 JSON Lines 的唯一出口。每次调用统一记录命令、项目根、阶段、诊断、变更、制品、消息、退出码和耗时；顶层入口捕获失败并先完成结构化结果，再保留非零进程状态。JSON Lines 的每个事件和最终结果都遵循版本 1 Schema，子进程失败还保留原始退出码。Schema 随源码和六平台发行档案一同提供。

## 审计门禁

核心工程协议负责验证锁定内容 SHA-256、远程来源、Git 精确与符号修订、SPDX 许可证、索引撤回/漏洞元数据、重复版本、目标架构及原生 ABI/目标/校验和/签名状态。远程索引、制品和 Git 来源在访问前只允许 HTTPS 或 SSH，下载先写同目录临时文件并原子替换缓存；本地路径和`file://`索引仍可用于离线与开发场景。索引把`yanked`和`vulnerabilities`作为显式可选元数据，字段缺失与空数组语义不同：缺失必须产生审计发现，不能推断为未撤回或无漏洞。

`src/审计.yx`负责验证核心发现对象、规范严重度、应用退出策略和复核抑制。每项发现必须是包含非空`code`、`severity`、`package`和`message`的有界对象；未知严重度、错误类型、缺失字段或不可安全用于抑制的代码都以`AUDIT_SCHEMA`拒绝继续。`warning`规范为`medium`，`error`规范为`high`，避免把无法理解的安全结果静默降级。

默认门禁从`high`开始，可按`critical`、`high`、`medium`、`low`、`info`或`none`配置。抑制按诊断代码精确匹配，漏洞代码包含规范化的公告标识，要求原因和 UTC 到期日，并在进入门禁前验证重复代码与日期。过期抑制不参与匹配，作为独立的`AUDIT_SUPPRESSION_EXPIRED`中等级发现报告。最终结果通过`audit_summary`公开计数和门禁配置；有效抑制通过`audit_suppressed`记录复核依据，任何未抑制且达到门禁的发现都会追加`AUDIT_DENIED`并保持非零退出。每类未抑制发现还提供对应的修复建议。

## 事务与失败语义

`new/init`使用基于规范目标摘要的父目录事务锁、同文件系统暂存目录、恢复点和原子替换日志。新目标在完整解析与静态检查后整体移动；已有目标只备份并替换清单、入口和锁文件，并且只在原先没有忽略文件时新建它。其他文件和目录不进入覆盖集。

`add/remove`在项目内取得跨调用事务锁，先记录原清单和原锁文件，再在事务目录内对同一工程请求执行清单预演。期望清单在真实编辑前写入日志；解析前后分别写入`resolving`和`resolved`阶段，解析成功后再记录期望锁文件。清单和锁文件的单次写入、同步与项目文件锁由言序包核心负责；言包日志负责把两次独立工程调用组成可恢复事务。

普通失败立即使用同目录临时文件和原子移动恢复原文；故障注入模拟的进程中断保留日志、恢复点和锁，之后必须显式使用`--recover`。恢复前会把当前文件与原文及日志中的期望事务内容比较；其他内容视为并发修改并以`TXN005`拒绝回滚。项目创建锁和项目事务锁互相排斥，其他言包项目命令在任一锁存在时以`TXN001`终止。工程协议错误是结构化 JSON，言包把错误转换为言序异常，因此进程退出码保持非零。

`标准:进程`受 `进程`权限控制，具备超时、UTF-8 校验和输出大小限制。`src/核心.yx`按操作类别选择超时，并在启动任何子进程前统一应用命令行或环境覆盖；超时后的终止与回收由言序运行时保证。源码启动器按用户明确调用在宿主权限下运行；若将言包编译为 YXB，权限来自 `言序.toml`。

## 安全边界

- 远程来源协议、制品解压、安全路径、校验和、原子缓存替换与压缩上限只由核心处理；
- 依赖只能沿预解析图访问，根包不能直接引用未声明的传递依赖；
- 原生扩展默认禁止，须显式权限和精确目标制品；
- 启动器不执行安装钩子或第三方构建脚本；
- 独立言包以宿主文件权限运行，但只对用户显式选择的工程执行文件操作；包内依赖与原生制品仍由本机言序的清单、锁定和校验门禁约束；
- 安装器只接受 GitHub Release 中与目标平台匹配且 SHA-256 校验通过的制品；
- 正式 Release 的 SBOM、逐目标构建元数据和证明必须绑定同一标签提交，不能由手工上传绕过；
- 未经安全评审的公开注册表写入不属于 0.2 发布范围。
