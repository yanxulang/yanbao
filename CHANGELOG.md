# 变更记录

## 0.6.1

### 审计与诊断

- `audit`在执行前验证工程协议握手，在结果返回后复验核心回显；只有 Schema 1 完整声明校验和、来源、精确 Git 修订、SPDX 许可证、撤回、漏洞、重复版本、目标及原生 ABI/制品检查时才接受结果，缺失、不完整或不一致均以`AUDIT_CAPABILITY_MISSING`失败关闭。
- 审计发现必须提供有界且合法的代码、严重度、包名和消息，可选文件及 1 基行号会进入 human、JSON 和 JSON Lines 诊断；畸形对象以`AUDIT_SCHEMA`拒绝，已知发现同时给出精确修复建议。
- 工程协议错误保留核心诊断代码、源码位置和子进程退出状态；空输出、坏 JSON、无效响应外壳与未知协议版本使用各自稳定错误，不再压缩成无法定位的通用失败。

### 供应链与发行

- 六个目标分别生成确定性的 CycloneDX 1.5 SBOM、构建元数据和依赖锁；元数据绑定言包源码提交、清单、锁、归档、独立应用及官方言序编译器标签提交，重复生成必须逐字节一致。
- 正式 Release 精确汇总 32 项资产：六组归档、SHA-256、构建元数据和锁，六份逐目标 SBOM，以及构建来源与 SBOM 证明；工作流先验验证全部发行主体、签名工作流、签名者、提交、来源引用、目标和摘要。
- 发布状态机只允许从主线创建不可移动标签，可恢复未完成的草稿或预发布，但不会回退、覆盖或降级已有稳定版；旧预发布补发时不会抢占更高稳定版本的 latest。
- 安装冒烟只消费先验验签后保存的本地只读制品，并在启动言序或言包子进程前移除 GitHub 下载凭据；归档内应用摘要和安装后的结构化版本都会再次核对。

### 兼容与安装

- 其他命令的最低运行时仍为言序 1.1.17；完整安全审计要求言序 1.1.18 提供十二项能力声明。清单格式 2、锁文件格式 2、工程协议 1、输出 Schema 1、YXB 与原生 ABI 均未改变，因此本版本按补丁发布。

macOS 与 Linux 可固定安装本版本：

```sh
curl -fsSL https://get.yanxu.dev/yanbao | YANBAO_VERSION=0.6.1 sh
```

Windows PowerShell 可先设置`$env:YANBAO_VERSION = "0.6.1"`，再运行
`irm https://get.yanxu.dev/yanbao/windows | iex`。

## 0.6.0

### 命令行与结构化接口

- 统一严格解析中英文子命令、位置参数、布尔旗标、单值与重复选项、互斥关系及`--`透传；未知参数、缺值、重复和冲突都以`CLI001..CLI010`稳定失败，不再回退到默认值。
- 所有命令支持 human、JSON 与 JSON Lines 输出，统一公开命令、项目根、阶段、诊断、变更、制品、耗时和退出状态；正式 Draft 2020-12 Schema 随六平台发行档案提供。
- 子进程超时按检查、解析、测试、构建、打包、网络和运行分别配置，`--timeout`、`--超时`与`YANBAO_TIMEOUT`支持 1 毫秒至 24 小时；最低运行时提升到言序 1.1.17，以保证长任务的终止和子进程回收。
- `audit`默认拒绝 critical/high，支持`critical|high|medium|low|info|none`门禁、可重复的精确代码抑制、原因与 UTC 到期日；过期配置、未知严重度和门禁失败分别返回稳定审计代码。`--deny-warnings`继续等价于 medium 门禁，`audit --json`正确保留非零失败状态。

### 事务与恢复

- `new/init`改为同文件系统暂存、完整解析与静态检查后提交；已有目标只替换清单、入口、锁文件及言包新建的忽略文件，`--force`不再删除无关内容。
- `add/remove`在真实写入前记录原清单、原锁文件和预演结果，覆盖解析失败、写入失败和中断后的分阶段恢复；普通失败立即回滚，中断现场须显式使用`--recover`或`--恢复`继续。
- 项目创建与依赖编辑共享独占锁和恢复协议，其他项目命令会拒绝进入未完成事务；并发篡改、无效或越界恢复日志均保留现场并以`TXN001..TXN005`拒绝覆盖。

### 发布与兼容

- 最低兼容、当前稳定和开发联调三通道继续按版本 JSON 与工程协议能力验收；安装器在下载前拒绝低于言序 1.1.17 的环境，正式制品仍不捆绑或覆盖言序运行时。
- 六目标 standalone 门禁新增参数错误、结构化输出、故障与中断恢复、并发拒绝和真实审计路径；两组事务规格可在独立进程中并行运行而不共享故障配置。
- Windows 校验文件固定为 ASCII 与单 LF，发布汇总同时使用`sha256sum`和`shasum`复验；构建、冒烟和发行说明从清单读取同一最低言序版本。
- 言窗项目在解析前通过核心工程协议将官方`yanxu-gui`依赖统一到`^1.0`；普通项目使用受严格 TLS 保护且固定 Release 摘要的只读索引，`--gui-path`保留本地联调，恢复日志记录修正后的清单。六目标制品准备统一使用工作区相对路径，避免 Windows 盘符路径被 Git Bash 重复拼接。
- 清单格式 2、锁文件格式 2、工程协议 1、YXB 格式和原生 ABI 要求不变；0.5.1 项目无需迁移。由于新增稳定命令参数、输出 Schema 和事务恢复能力，本版本按次版本发布。

### 安装

macOS 与 Linux 可固定安装本版本：

```sh
curl -fsSL https://get.yanxu.dev/yanbao | YANBAO_VERSION=0.6.0 sh
```

Windows PowerShell 可先设置`$env:YANBAO_VERSION = "0.6.0"`，再运行
`irm https://get.yanxu.dev/yanbao/windows | iex`。

## 0.5.1

### 三通道工具链兼容

- 最低兼容通道从`言序.toml`读取`>=1.1.7`并固定构建`v1.1.7`，持续证明言包没有无意依赖更高版本。
- 当前稳定通道动态查询言序最新正式 Release，通过公开安装器使用真实平台制品；Linux、macOS、Windows 的 x86-64 与 ARM64 验收继续覆盖独立应用、启动器、言窗工程和 Bundle。
- 开发联调通道构建言序`main`，读取`yanxu version --json`和工程协议握手，按清单、锁、YXB、原生 ABI 与操作能力判定兼容，不断言固定开发版本字符串。
- 言包规格不再把工具链版本写死为 1.1.7，而是核对同一运行时的版本报告和能力握手；版本 JSON 与工程协议不一致会立即失败。

### 兼容与升级

- 最低支持言序仍为 1.1.7，清单格式 2、锁文件格式 2、工程协议 1 和原生 ABI v1/v2 要求不变。
- 不改变现有命令、参数、输出或项目文件；0.5.0 用户可直接升级，既有清单和锁文件无需迁移。
- 正式制品仍只包含言包独立应用和启动器，不捆绑或覆盖用户的言序运行时。

### 安装

macOS 与 Linux 可固定安装本版本：

curl -fsSL https://get.yanxu.dev/yanbao | YANBAO_VERSION=0.5.1 sh

## 0.5.0

### 言序 1.1.7 GUI 工程

- 工具链基线升级到言序 1.1.7，并要求原生 ABI v1/v2 能力握手。
- `yanbao new/init --gui`通过共享工程协议生成官方言窗清单、权限和窗口源码。
- `yanbao build --release --bundle`和`yanbao bundle`生成当前平台桌面应用。
- `build --release --gui`、`build --图形`和中文`应用包`使用同一 Bundle 流程。
- 六目标 CI 在 Linux、macOS、Windows 的 x86-64 与 ARM64 原生机上构建言窗后端并验收 GUI 工程。

### 持续集成

- GUI 集成期间从匹配的核心分支构建 1.1.7 工具链；合并后回到核心`main`。

## 0.4.0

### GitHub 默认包源

- `yanbao add <包名>` 默认解析到 `https://github.com/yanxulang/yanxu-<包名>.git`，无需维护中央包服务器。
- `yanxu-` 包名前缀和 `yanxulang/` 组织名均可省略；也支持 `<组织>/<仓库>` 引用第三方 GitHub 包。
- 本地依赖别名自动省略 `yanxu-`，同时保留 `--package` 覆盖以及显式 `--git`、`--path`、`--registry` 来源。
- GitHub 简写解析具有独立规格测试，原有清单与锁文件事务回滚保持不变。

## 0.3.1

### Windows 兼容性

- 安装器和 PowerShell 启动器兼容系统自带的 Windows PowerShell 5.1，不再依赖 PowerShell 7 语法。
- Windows CMD 启动器改为通过兼容层调用 PowerShell，并增加 ASCII、CRLF 与双 PowerShell 解析门禁。
- Release 的 Windows 构建和安装冒烟使用修正后的启动链重新验证六平台制品。

## 0.3.0

### 运行时依赖

- 正式发行包不再捆绑专用言序运行时，统一复用`YANXU_BIN`或`PATH`中的本机言序 1.1.6 或更高版本。
- Unix、PowerShell 与 CMD 启动器在言序缺失时立即给出明确提示并退出；安装器在下载制品前执行同样的环境检查。
- 六平台 Release 仍提供编译后的 standalone 言包应用，但归档只包含言包启动器、应用和文档。

### 发布安全

- Release 在平台封装和最终上传两个阶段检查文件清单，禁止任何`yanxu`运行时进入发行制品。
- 安装冒烟显式准备外部言序环境，再验证言包安装、版本握手和工程协议。
- 发行说明明确列出言序 1.1.6 或更高版本这一前置条件。

## 0.2.1

### 言序 1.1.6 基线

- 最低工程协议与构建运行时升级到言序 1.1.6，继续保持命令和包管理业务 100% 使用言序源码实现，不引入 Cargo/Rust 工程。
- 正式 Release 提供六目标独立应用和专用言序 1.1.6 工程运行时；用户无需预装言序或 Rust，即可完成初始化、依赖、测试、构建、vendor 与审计流程。
- CI 统一使用核心`v1.1.6`标签，不再把行为不同的裸提交继续称为 1.1.5。

### 发布一致性

- 自动发布改为显式触发并先运行完整三平台质量门禁；既有版本标签不再允许强制移动。
- 标签必须精确指向`origin/main`，六平台本机 standalone 构建和工程端到端测试全部通过后才上传 Release 草稿，12 个归档/校验文件齐全后才公开。
- 安装器继续强制 SHA-256、临时安装和版本验证；公开后在 Linux、macOS、Windows 重新执行指定版本安装冒烟，失败时把 Release 转回草稿隔离。

## 0.2.0

### 重大变更

- 删除 Cargo 工程和全部 Rust 业务代码，使用言序重写命令层。
- 最低运行环境提升为言序 1.1.5；用户无需 Rust，也无需预编译言包。
- 通过版本化 JSON 工程协议复用唯一的清单格式 2、锁文件格式 2、传递依赖图和 YXB 构建语义。

### 新增

- 新增 `check/test/build/clean/tree/why/doctor/doc` 及中文命令。
- 新增 YXB 和当前平台独立应用构建。
- 新增 Unix、PowerShell 与 CMD 纯启动器和三平台纯言序 CI。
- 依赖变更继续保持清单/锁文件事务回滚。
- 新增由言序 1.1.5 编译的 Linux、macOS、Windows x86-64/ARM64 独立制品，并捆绑专用 1.1.5 工程运行时。
- 构建和捆绑运行时固定使用上游`v1.1.5-fix`发行制品；该修正版的程序版本仍为言序 1.1.5。
- 新增带 SHA-256 强制校验的 Unix 与 Windows 安装脚本，以及版本、标签、主分支、制品和安装冒烟门禁。

## 0.1.0

### 新增

- 新增`init`、`add`、`remove`、`install`、`update`、`list`和`run`，并提供中文别名。
- 支持路径、Git 和中央索引依赖，解析与锁文件生成复用`yanxu::package`。
- 新增离线安装、全局清单定位、稳定 JSON 输出和依赖变更事务回滚。
- 新增单元/端到端测试、三平台 CI、架构与路线图文档。
- 清单模板同步言序 1.1.3 的`TCP监听`与`UDP绑定`权限，避免把外连权限误用为本地监听授权。
- 核心依赖升级到`yanxu v1.1.4-A`，`run`同步获得字节串、二进制 I/O、安全密码原语、命令行参数与锁定包模块加载支持。
- `run -- <参数...>`会把分隔符后的参数原样交给包入口。
- 配合核心包模块授权后，Git 依赖无需 submodule 或把全局缓存目录写入应用权限；显式更新会刷新符号修订并继续锁定精确提交。

### 安全

- 依赖核心锁定到`yanxu v1.1.4-A`，继承安全制品解压、真实 IP 复查、套接字资源限制、有界二进制读取、持久 Engine 一致性与锁定包权限修复。
